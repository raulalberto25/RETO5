**ADR 0001 Example Structure:**
```
# ADR 0001: Strangler Fig Pattern for Monolith Migration

## Context
Large banking monolith with 50+ services. Need to extract services incrementally
without requiring full system rewrite. Production system cannot afford downtime.

## Decision
Adopt Strangler Fig pattern:
1. New API Gateway routes requests
2. Gateway routes known endpoints to new services
3. Unknown endpoints routed to monolith (legacy)
4. Gradually extract services as they mature
5. Eventually monolith becomes empty and is decommissioned

## Alternatives Considered
1. Big Bang Rewrite
   - Pros: Clean slate, optimal design
   - Cons: High risk, requires 6+ months, business cannot add features

2. Feature Branching
   - Pros: Simpler locally
   - Cons: Merge conflicts, doesn't solve distributed nature

3. Anti-Corruption Layer (Strangler alternative)
   - Pros: Flexible boundary
   - Cons: More translation layers, higher latency

## Consequences
- Pros:
  * Incremental migration, low risk
  * Services can be extracted in priority order
  * Existing consumers see no changes
  * Can run parallel A/B tests
  
- Cons:
  * Gateway adds latency
  * Data duplication during migration
  * Complex eventual consistency
  * Monitoring more difficult

## Implementation Details
- Gateway: YARP (low latency, .NET-native)
- Routing: URL-based (simple, observable)
- Traffic shifting: Canary deployment (Phase 4)
- Rollback: Route back to monolith (instant)

## Compliance & Regulatory Notes
- Audit trail: Correlation ID + Kafka event log
- Data consistency: Eventual consistency documented (ADR 0006)
- Bank regulations: No service downtime requirement enforced
```

```csharp
public class CreateTransactionSaga : StateMachine<CreateTransactionSagaState>
{
    // Define states
    public State Started { get; private set; }
    public State SourceAccountValidated { get; private set; }
    public State DestinationAccountValidated { get; private set; }
    public State BalanceReserved { get; private set; }
    public State TransferInitiated { get; private set; }
    public State SourceDebited { get; private set; }
    public State DestinationCredited { get; private set; }
    public State TransactionCompleted { get; private set; }
    public State Failed { get; private set; }
    public State Compensating { get; private set; }
    
    // Define events (from Kafka or direct commands)
    public Event<CreateTransactionCommand> CreateTransactionRequested { get; private set; }
    public Event<SourceAccountValidatedEvent> SourceAccountValidated { get; private set; }
    // ... more events
    
    // Idempotency: Track correlation ID + step
    public override void Configure()
    {
        // Initially
        Initially(
            When(CreateTransactionRequested)
                .Activity(x => x.OfType<ValidateSourceAccountActivity>())
                .TransitionTo(SourceAccountValidated)
        );
        
        // From SourceAccountValidated
        During(SourceAccountValidated,
            When(SourceAccountValidatedEvent)
                .Activity(x => x.OfType<ValidateDestinationAccountActivity>())
                .TransitionTo(DestinationAccountValidated),
            When(SourceAccountValidationFailedEvent)
                .Activity(x => x.OfType<CompensationActivity>())
                .TransitionTo(Failed)
        );
        
        // ... more transitions
        
        // Compensation chain for failures
        During(Failed,
            When(CompensationCompleted)
                .Finalize()
        );
    }
}
```

```csharp
public class ValidateSourceAccountActivity : Activity<CreateTransactionSagaState, SourceAccountValidatedEvent>
{
    private readonly IAccountServiceClient _accountServiceClient;
    private readonly ILogger<ValidateSourceAccountActivity> _logger;
    
    public async Task<CompensationResult<CreateTransactionSagaState>> Compensate(
        CompensationContext<CreateTransactionSagaState> context)
    {
        // No compensation needed for validation-only step
        return context.Compensated();
    }
    
    public async Task Execute(
        ExecuteContext<CreateTransactionSagaState, SourceAccountValidatedEvent> context)
    {
        try
        {
            var sourceAccount = await _accountServiceClient.GetAccountAsync(
                context.Instance.SourceAccountId,
                context.Instance.CorrelationId);
                
            if (sourceAccount == null)
                throw new AccountNotFoundException($"Source account {context.Instance.SourceAccountId} not found");
            
            // Publish event for saga to continue
            await context.Publish(new SourceAccountValidatedEvent
            {
                CorrelationId = context.Instance.CorrelationId,
                AccountId = sourceAccount.Id,
                Balance = sourceAccount.Balance
            });
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Source account validation failed for transaction {TransactionId}", 
                context.Instance.TransactionId);
                
            await context.Publish(new SourceAccountValidationFailedEvent
            {
                CorrelationId = context.Instance.CorrelationId,
                Reason = ex.Message
            });
        }
    }
}
```

#### Step 2.3.2: Event Publishing Guarantees

**Files to create:**
- `src/Services/TransactionService/Infrastructure/Outbox/OutboxEntry.cs`
- `src/Services/TransactionService/Infrastructure/Outbox/OutboxPublisher.cs`

**At-Least-Once Delivery Pattern:**

```
Problem: Service publishes event to Kafka, then crashes before committing.
Solution: Outbox pattern (also called transactional outbox)

Implementation:
1. Save domain event to Outbox table (same transaction as domain state)
2. Commit transaction (both outbox entry and domain state in DB)
3. Separate background process polls Outbox
4. Publishes entries to Kafka
5. Marks entry as published
6. If Kafka publish fails, entry remains in Outbox (retry later)

Database schema:
CREATE TABLE Outbox (
    Id BIGINT PRIMARY KEY IDENTITY,
    AggregateId GUID NOT NULL,
    EventType NVARCHAR(255) NOT NULL,
    EventData NVARCHAR(MAX) NOT NULL,  -- JSON
    CreatedAt DATETIME2 NOT NULL,
    PublishedAt DATETIME2 NULL,  -- NULL until published
    Attempts INT DEFAULT 0
);

Background process (OutboxPublisher):
- Query: SELECT * FROM Outbox WHERE PublishedAt IS NULL ORDER BY CreatedAt
- For each entry:
  * Publish to Kafka
  * If success: UPDATE Outbox SET PublishedAt = NOW()
  * If failure: Increment Attempts, retry exponentially

Guarantees:
- At-least-once delivery (Kafka sees events multiple times possible)
- Exactly-once processing by consumers (requires idempotency keys)
- No lost events (stored in DB)
- Works even if service crashes
```

```csharp
// Use case: Transaction failed at step 5 (debit source account failed)

public class ImmediateReversalPolicy
{
    public async Task HandleFailureAsync(
        FailureContext context,
        Step previousStep)
    {
        if (context.Exception is InsufficientFundsException)
        {
            // Money never left account, no reversal needed
            return;

        if (previousStep.Type == StepType.Debit)

                accountId: context.SourceAccountId,
                amount: context.Amount,
                reason: "Transaction reversal due to " + context.Exception.Message,
                originalTransactionId: context.TransactionId);
            
            // Mark transaction as reverted in database
            await transactionService.MarkRevertedAsync(context.TransactionId);
        }
    }
}
```

```csharp
// Pattern: Retry with Exponential Backoff for Transient Failures
// Use case: Account Service temporarily unavailable (503)
// Action: Retry up to 3 times with exponential backoff

var policy = Policy.Handle<HttpRequestException>()
    .Or<OperationCanceledException>()
    .OrResult<HttpResponseMessage>(r => r.StatusCode == HttpStatusCode.ServiceUnavailable)
    .WaitAndRetryAsync(
        retryCount: 3,
        sleepDurationProvider: attempt => TimeSpan.FromMilliseconds(
            100 * Math.Pow(2, attempt - 1) +  // Exponential: 100ms, 200ms, 400ms
            Random.Next(0, 50)  // Jitter: 0-50ms
        ),
        onRetry: (outcome, timespan, attempt, context) =>
        {

                "Retry attempt {Attempt} after {Delay}ms for transactionId={TransactionId}",
                attempt, timespan.TotalMilliseconds, context["transactionId"]);
        });
```

```csharp
// Pattern: Circuit Breaker to Prevent Cascading Failures

var circuitBreakerPolicy = Policy.Handle<HttpRequestException>()
```

**Pattern 4: Idempotency Keys for Exactly-Once Processing**

```csharp
// Pattern: Idempotency Keys to Prevent Duplicate Transactions
// Use case: Client retries transaction after timeout (doesn't know if first request succeeded)
// Action: Server remembers idempotency key, returns cached result on retry

public class IdempotencyMiddleware
{
    public async Task InvokeAsync(HttpContext context)
    {
        // Check Idempotency-Key header
        if (context.Request.Headers.TryGetValue("Idempotency-Key", out var idempotencyKey))
        {
            var cachedResponse = await idempotencyStore.GetAsync(idempotencyKey);
            if (cachedResponse != null)
            {
                // Return cached response (same status, same body)
                context.Response.StatusCode = cachedResponse.StatusCode;
                await context.Response.WriteAsJsonAsync(cachedResponse.Body);
                return;
            }
        }
        
        // Execute request normally
        var originalBody = context.Response.Body;
        using var memoryStream = new MemoryStream();
        context.Response.Body = memoryStream;
        
        await next(context);
        
        // Cache the response
        memoryStream.Position = 0;
        var responseBody = await new StreamReader(memoryStream).ReadToEndAsync();
        
        if (idempotencyKey != StringValues.Empty)
        {
            await idempotencyStore.SetAsync(
                idempotencyKey,
                new IdempotencyEntry
                {
                    StatusCode = context.Response.StatusCode,
                    Body = responseBody,
                    CreatedAt = DateTime.UtcNow
                },
                expirationTime: TimeSpan.FromHours(24));  // Banking requirement: 24h minimum
        }
        
        // Return response
        memoryStream.Position = 0;
        await memoryStream.CopyToAsync(originalBody);
    }
}
```

```csharp
public class AccountEventConsumer : IConsumer<AccountCreatedEvent>
{
    private readonly ITransactionRepository _repository;
    private readonly ILogger<AccountEventConsumer> _logger;
    
    public async Task Consume(ConsumeContext<AccountCreatedEvent> context)
    {
        var @event = context.Message;
        
        // Idempotency: Check if we've processed this event before
        var processedEvent = await _repository.GetProcessedEventAsync(
            @event.EventId, 
            @event.AggregateId);
            
        if (processedEvent != null)
        {
            _logger.LogInformation(
                "Event already processed: {EventId}. Skipping.",
                @event.EventId);
            return;  // Idempotent (no-op if already processed)
        }
        
        try
        {
            // Update local state based on event
            await _repository.RegisterNewAccountAsync(
                @event.AccountId,
                @event.CustomerId,
                @event.AccountType,
                @event.Currency);
            
            // Mark event as processed
            await _repository.MarkEventProcessedAsync(@event.EventId, @event.AggregateId);
            
            _logger.LogInformation(
                "Account {AccountId} registered in TransactionService. CorrelationId={CorrelationId}",
                @event.AccountId, @event.CorrelationId);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex,
                "Failed to process AccountCreatedEvent {EventId}. " +
                "Message will be retried.",
                @event.EventId);
            
            // Throwing exception causes MassTransit to retry
            throw;
        }
    }
}

public class EventConsumerConfiguration
{
    public static void AddEventConsumers(this IServiceCollection services)
    {
        services.AddMassTransit(x =>
        {
            x.AddConsumer<AccountEventConsumer>();
            x.AddConsumer<AccountSuspendedEventConsumer>();
            
            x.AddBus(provider => Bus.Factory.CreateUsingKafka((cfg, k) =>
            {
                k.Host(kafkaSettings.BootstrapServers);
                
                cfg.ReceiveEndpoint("transaction-service-account-events", e =>
                {
                    e.ConfigureConsumer<AccountEventConsumer>(provider);
                    
                    // Retry policy
                    e.UseDelayedRedelivery(r => r.Intervals(100, 1000, 5000));
                    
                    // Dead letter queue
                    e.UseInMemoryOutbox();
                });
            }));
        });
    }
}
```

### 3.2 Consistency Guarantees

#### Step 3.2.1: Eventual Consistency Documentation

**Files to create:**
- `docs/ADR/0006-eventual-consistency.md`

**Consistency Models Explained:**

```
Model 1: Strong Consistency (Not used in this system)
- All services always see same data
- Impossible in distributed systems without sacrificing availability
- Not suitable for banking with high availability requirements

Model 2: Eventual Consistency (Primary pattern)
- Services see different data temporarily
- Given no new updates, all services converge to same state
- Banking implication: Customer account balance might be slightly out of sync
  between Account Service and Transaction Service for <5 seconds
- Acceptable because: No money is lost, regulations allow eventual consistency
  for non-critical operations

Model 3: Causal Consistency (For critical operations)
- Events are processed in causal order (if E1 caused E2, E2 processed after E1)
- Implemented via: Saga pattern with explicit ordering
- Example: TransactionInitiated must be processed before TransactionCompleted

Guarantee for this system:
1. Within a single aggregate (Account, Transaction): Strong consistency
   - Database ACID guarantees
   - Single writer principle
   
2. Between aggregates (Account + Transaction consistency):
   Eventual consistency via Saga
   - Saga ensures logical ordering
   - Both accounts eventually reconcile
   - Worst case: 5 second lag
   
3. For fraud detection, audit logging:
   Strict ordering via single-partition topic
   - Topic with 1 partition guarantees order
   - Slightly lower throughput but strict compliance
```

```csharp
public class TransactionProjection
{
    // Denormalized view, optimized for queries
    public Guid TransactionId { get; set; }
    public Guid SourceAccountId { get; set; }
    public Guid DestinationAccountId { get; set; }
    public string SourceAccountNumber { get; set; }  // Denormalized from Account Service
    public string DestinationAccountNumber { get; set; }
    public decimal Amount { get; set; }
    public string Currency { get; set; }
    public string Status { get; set; }
    public DateTime CreatedAt { get; set; }
    public DateTime? CompletedAt { get; set; }
}

public class TransactionProjectionHandler :
    IConsumer<TransactionInitiatedEvent>,
    IConsumer<TransactionCompletedEvent>,
    IConsumer<TransactionFailedEvent>
{
    private readonly IProjectionRepository _repository;
    
    public async Task Consume(ConsumeContext<TransactionInitiatedEvent> context)
    {
        var @event = context.Message;
        
        // Create initial projection
        var projection = new TransactionProjection
        {
            TransactionId = @event.TransactionId,
            SourceAccountId = @event.SourceAccountId,
            DestinationAccountId = @event.DestinationAccountId,
            Amount = @event.Amount,
            Currency = @event.Currency,
            Status = "Pending",
            CreatedAt = @event.Timestamp
        };
        
        await _repository.CreateProjectionAsync(projection);
    }
    
    public async Task Consume(ConsumeContext<TransactionCompletedEvent> context)
    {
        var @event = context.Message;
        
        // Update projection
        var projection = await _repository.GetByIdAsync(@event.TransactionId);
        projection.Status = "Completed";
        projection.CompletedAt = @event.Timestamp;
        
        await _repository.UpdateProjectionAsync(projection);
    }
}

// Query usage
public class GetTransactionHistoryQueryHandler : IRequestHandler<GetTransactionHistoryQuery>
{
    private readonly IProjectionRepository _repository;
    
    public async Task<List<TransactionProjection>> Handle(GetTransactionHistoryQuery request)
    {
        return await _repository
            .Where(p => p.SourceAccountId == request.AccountId || 
                        p.DestinationAccountId == request.AccountId)
            .OrderByDescending(p => p.CreatedAt)
            .ToListAsync();
    }
}
```

```markdown
# ADR 0005: Saga Pattern Orchestration

## Status
Accepted

## Context
Need to maintain consistency across Account and Transaction services during
transaction processing. Two options: choreography vs orchestration.

Choreography: Services publish events, other services consume and react
- Pros: Decoupled, no central coordinator
- Cons: Complex to follow, no single view of saga state, hard to debug

Orchestration: Central saga orchestrator sends commands to services
- Pros: Explicit workflow, single source of truth, easy to monitor
- Cons: Central point of failure, harder to scale

## Decision
Use hybrid approach:
- Simple flows (creating account): Use choreography
- Complex flows (transaction with compensation): Use orchestration

Saga orchestrator implemented using MassTransit StateMachine.

## Consequences
- Positive: Clear transaction semantics, easier troubleshooting
- Negative: Extra latency from orchestrator, need to persist saga state

## Compliance Notes
- Saga state persisted to database (durability required by banking)
- Compensation logic tested for all failure scenarios (audit trail)
- Idempotency keys prevent duplicate transactions
```

## Technology Decisions & Alternatives Analysis

### 0A. API Gateway: YARP vs Kong vs Nginx vs Envoy

| Criteria | YARP | Kong | Nginx | Envoy |
|----------|------|------|-------|-------|
| Language | C# (.NET) | Lua/Go | C | Go |
| .NET Integration | Native | Limited | Limited | Limited |
| Latency | Ultra-low | Low | Ultra-low | Low |
| Configuration | C# code | YAML/API | nginx.conf | YAML |
| Features | Basic | Rich (plugins) | Rich | Rich |
| Learning Curve | Low (.NET devs) | Medium | Medium | High |
| **Decision** | **Selected** | Alternative | Alternative | Enterprise alternative |

**Rationale:** YARP chosen for native .NET integration, low latency, and easy configuration from C# code. Reduces operational overhead in .NET-heavy organization.

### 0B. Message Broker: Kafka vs RabbitMQ

| Criteria | Kafka | RabbitMQ |
|----------|-------|----------|
| Throughput | High (1M msg/sec) | Medium (50K msg/sec) |
| Latency | Medium (10ms+) | Low (sub-ms) |
| Durability | Excellent (event log) | Good |
| Ordering | Per-partition | Per-queue |
| Compliance | Audit trail built-in | Manual implementation |
| Operational Complexity | Higher | Lower |
| .NET Support | Excellent | Excellent |
| **Decision** | **Selected** | Alternative |

**Rationale:** Kafka selected for immutable event log (critical for banking compliance), topic partitioning (parallelism), and native ordering guarantees. The slightly higher operational complexity is acceptable.

### 0C. Saga Pattern: Orchestration vs Choreography

| Aspect | Orchestration | Choreography |
|--------|---------------|-------------|
| Complexity | Medium | High |
| Visibility | Excellent (single orchestrator) | Poor (distributed logic) |
| Debugging | Easy | Difficult |
| Performance | Slightly slower | Slightly faster |
| Failure Handling | Centralized | Distributed |
| **Decision** | **Selected (primary)** | Fallback for simple flows |

**Rationale:** Hybrid approach selected. Use orchestration for complex banking workflows (transactions), choreography for simple events (account creation notifications).

### 0D. Database: PostgreSQL vs SQL Server vs MySQL

| Criteria | PostgreSQL | SQL Server | MySQL |
|----------|-----------|-----------|-------|
| ACID Compliance | Strict | Strict | Configurable |
| JSON Support | Excellent | Good | Good |
| Licensing | Open source | Paid | Open source |
| .NET Support | Excellent | Native | Good |
| Operational | Easy | Medium | Easy |
| **Decision** | **Selected** | For existing orgs | Not recommended |

**Rationale:** PostgreSQL selected for open-source licensing, excellent JSON support (for outbox pattern), and strong ACID guarantees needed for financial data.

## Critical Technical Challenges & Mitigation

### Challenge 1: Data Consistency Across Services

**Problem:** Account service has $100, transaction service thinks $100, but in reality we debit $50. Services out of sync.

**Mitigation:**
1. Saga pattern ensures steps executed in order
2. Idempotency keys prevent duplicate executions
3. Eventual consistency documented and accepted
4. Daily reconciliation job compares Accountâ†”Transaction balances
5. Monitoring alerts on imbalance > $0.01

**Implementation:**
```csharp
// Daily reconciliation job
public class AccountTransactionReconciliationJob : IHostedService
{
    public async Task ExecuteAsync()
    {
        var accountBalances = await accountService.GetAllBalancesAsync();
        var transactionBalances = await transactionService.GetAllBalancesAsync();
        
        var discrepancies = accountBalances
            .Join(transactionBalances, a => a.AccountId, t => t.AccountId,
                  (a, t) => new { Account = a, Transaction = t })
            .Where(x => Math.Abs(x.Account.Balance - x.Transaction.Balance) > 0.01m)
            .ToList();
        
        if (discrepancies.Any())
        {
            await alertingService.SendAlertAsync(
                severity: AlertSeverity.Critical,
                message: $"{discrepancies.Count} account balance discrepancies detected");
            
            // Create reconciliation records for manual review
            foreach (var discrepancy in discrepancies)
            {
                await reconciliationRepository.CreateAsync(
                    accountId: discrepancy.Account.AccountId,
                    expectedBalance: discrepancy.Account.Balance,
                    actualBalance: discrepancy.Transaction.Balance,
                    difference: discrepancy.Account.Balance - discrepancy.Transaction.Balance);
            }
        }
    }
}
```

### Challenge 2: Saga Failure & Compensation Complexity

**Problem:** Transaction saga fails at step 5 (debit source). Must reverse steps 4, 3, 2, 1. If reversal fails, saga stuck forever.

**Mitigation:**
1. Compensation logic is itself idempotent (can be retried)
2. Saga state persisted to database (can resume from failure)
3. Dead letter queue captures unprocessable compensation commands
4. Manual intervention playbook for stuck sagas

**Implementation:**
```csharp
public class SagaCompensationActivity : Activity
{
    public async Task<CompensationResult<CreateTransactionSagaState>> Compensate(
        CompensationContext<CreateTransactionSagaState> context)
    {
        // Compensation is idempotent
        // If called twice, same result
        
        // Step 1: Release source account reservation
        var releaseResult = await accountService.ReleaseReservationAsync(
            accountId: context.Instance.SourceAccountId,
            reservationId: context.Instance.SourceAccountReservationId);
        
        if (!releaseResult.Success && releaseResult.Error != "ReservationNotFound")
        {
            // Retry this step
            return context.Retry();  // MassTransit retries
        }
        
        // Step 2: Mark transaction as failed
        await transactionService.MarkFailedAsync(
            transactionId: context.Instance.TransactionId,
            reason: "Saga compensation: " + context.Exception?.Message);
        
        return context.Compensated();  // Compensation complete
    }
}
```

### Challenge 3: Network Partitions & Service Failures

**Problem:** Account Service is down. Transaction service can't validate accounts. Should fail immediately or retry?

**Mitigation:**
1. Circuit breaker stops sending requests after 5 failures
2. Saga fails fast (doesn't wait forever)
3. Fallback: Use last-known-good data from cache
4. Manual intervention: Operations team can manually approve transaction

**Implementation:**
```csharp
// Circuit breaker + fallback policy
var policy = Policy.Handle<HttpRequestException>()
    .OrResult<AccountResponse>(r => r == null)
    .CircuitBreakerAsync(
        handledEventsAllowedBeforeBreaking: 5,
        durationOfBreak: TimeSpan.FromSeconds(30))
    .WrapAsync(
        Policy.TimeoutAsync<AccountResponse>(TimeSpan.FromSeconds(5))
    )
    .FallbackAsync<AccountResponse>(
        async (outcome, context) =>
        {
            // Fallback: Use cached account data
            var cachedAccount = await cache.GetAsync(
                $"account:{context["accountId"]}");
            return cachedAccount;  // May be stale, but better than failure
        });
```

### Challenge 4: Regulatory Compliance & Audit Trails

**Problem:** Bank regulators audit our transactions. Must prove every transaction was processed correctly.

**Mitigation:**
1. All events stored in Kafka (immutable append-only log)
2. Correlation IDs tie requests to events
3. All service calls logged with timestamps
4. Saga state transitions audited
5. Annual compliance audits validate log completeness

**Implementation:**
```csharp
// Outbox ensures event logged before returning to client
public class TransactionCommandHandler : IRequestHandler<CreateTransactionCommand>
{
    private readonly UnitOfWork _unitOfWork;
    
    public async Task<TransactionResponse> Handle(CreateTransactionCommand command)
    {
        using var transaction = await _unitOfWork.BeginTransactionAsync();
        
        // Create transaction in database
        var transaction = new Transaction
        {
            Id = command.TransactionId,
            CorrelationId = command.CorrelationId,  // Link to request
            SourceAccountId = command.SourceAccountId,
            DestinationAccountId = command.DestinationAccountId,
            Amount = command.Amount,
            Status = TransactionStatus.Initiated,
            CreatedAt = DateTime.UtcNow,
            CreatedBy = currentUser.Id
        };
        await _unitOfWork.Transactions.AddAsync(transaction);
        
        // Add event to outbox (same transaction)
        var @event = new TransactionInitiatedEvent
        {
            EventId = Guid.NewGuid(),
            TransactionId = command.TransactionId,
            CorrelationId = command.CorrelationId,
            Timestamp = DateTime.UtcNow
        };
        await _unitOfWork.Outbox.AddAsync(@event);
        
        // Single atomic commit
        await _unitOfWork.SaveChangesAsync();  // Commits both transaction + event
        
        // Only after commit, return response
        return new TransactionResponse { Id = transaction.Id };
        
        // If service crashes here, event is still in Outbox
        // Background process will publish it later
    }
}
```

### Challenge 5: Schema Evolution & Versioning

**Problem:** We want to add "description" field to AccountCreatedEvent. Old services don't understand new field.

**Mitigation:**
1. New fields always optional with defaults
2. Old services ignore unknown fields (schema-agnostic deserialization)
3. Separate versioning for breaking changes (v2 endpoint)
4. Deprecation period: 6 months before removing old versions

**Implementation:**
```csharp
// Event deserialization with unknown fields ignored
public class AccountCreatedEvent
{
    [JsonPropertyName("accountId")]
    public Guid AccountId { get; set; }
    
    [JsonPropertyName("customerId")]
    public Guid CustomerId { get; set; }
    
    // New field: optional, defaults to null
    [JsonPropertyName("description")]
    [JsonIgnore(Condition = JsonIgnoreCondition.WhenWritingNull)]
    public string? Description { get; set; }
}

// Deserializer that ignores unknown fields
var options = new JsonSerializerOptions
{
    PropertyNameCaseInsensitive = true,
    DefaultIgnoreCondition = JsonIgnoreCondition.WhenWritingNull,
    WriteIndented = false
};

// Unknown fields are silently ignored
var @event = JsonSerializer.Deserialize<AccountCreatedEvent>(json, options);
```

🔄 Flujo de Publicación (Outbox Pattern)

1. Transfers MS ├─ TransferUseCase.ExecuteAsync() ├─ Crea Transfer aggregate ├─ Crea TransferExecutedEvent (CloudEvents) ├─ Guarda Transfer + OutboxEntry (MISMA transacción) └─ Responde 201 al cliente 2. OutboxWorker (Background Service) ├─ Cada 5 segundos: │ ├─ Lee OutboxEntries (published_at IS NULL) │ ├─ Publica a RabbitMQ exchange=banking.events │ └─ Marca published_at = NOW() └─ Si falla: reintenta automáticamente 3. RabbitMQ ├─ Topic Exchange: banking.events └─ Enruta a colas que coincidan con patrón 4. Consumers ├─ NotificationsConsumer: crea notificación └─ AuditConsumer: crea audit entry

✅ Beneficios de la Arquitectura

  • Desacoplamiento: Transfers no conoce a consumers
  • Escalabilidad: Agregar consumers sin cambiar Transfers
  • Garantía de entrega: Outbox = at-least-once semantics
  • Resiliencia: Consumer falla no afecta otros
  • Traceabilidad: CloudEvents con correlationId

Save Transfer + OutboxEntry (atomic) ↓ 201 Created (inmediato) ↓ [5s later] OutboxWorker publica evento ↓ NotificationsConsumer crea notificación AuditConsumer crea audit entry RESULTADO: ✅ Transfer (STRONG) + Notifications/Audit (EVENTUAL)

❌ Failure Path 1: Accounts MS No Responde

GET /accounts/X1 → TIMEOUT ↓ RETRY 1 (wait 1s) → TIMEOUT RETRY 2 (wait 2s) → TIMEOUT RETRY 3 (wait 4s) → TIMEOUT ↓ CIRCUIT BREAKER OPENS (fail fast) ↓ 503 Service Unavailable (no Transfer creado) RESULTADO: ❌ Sin Transfer + Sin eventos + CB previene cascada

❌ Failure Path 2: RabbitMQ Se Cae

Transfer + OutboxEntry guardados ✅ 201 Created enviado ✅ ↓ [5s] OutboxWorker intenta publicar → CONNECTION FAILED OutboxEntry aún tiene published_at=NULL ↓ [10s] OutboxWorker reintenta RabbitMQ recuperado Evento publicado exitosamente ✅ ↓ Consumers procesan normalmente RESULTADO: ✅ Transfer + Garantía de entrega (delayed)

❌ Failure Path 3: Consumer Falla

Evento publicado ✅ NotificationsConsumer recibe ↓ SendAsync() → DATABASE ERROR BasicNack() → Requeue ↓ RabbitMQ redelivery (con delay) Segundo intento → Falla nuevamente ↓ BasicNack() → Move to DLQ (Dead Letter Queue) RESULTADO: ✅ Transfer + Audit (OK) + Notification (DLQ) = Degradación

3️⃣ Patrones de Resiliencia (3 Patrones Funcionales)

🛡️ Patrón 1: Circuit Breaker

Ubicación: ResiliencePolicies.cs

Configuración: 5 failures en 30s → open, 30s break duration

CLOSED (normal) ↓ 5 failures OPEN (fail-fast) ↓ 30 seconds HALF-OPEN (probe) ↓ probe succeeds CLOSED (reset)

Beneficios: Previene cascada, fail-fast, recuperación automática

🔄 Patrón 2: Retry con Backoff Exponencial

Configuración: 3 intentos, 2^attempt segundos

Attempt 1 (immediate) Attempt 2 (wait 2s) Attempt 3 (wait 4s) Attempt 4 (wait 8s) [5 failures = Circuit Breaker activation]

Beneficios: Maneja fallos transitorios, reduce carga

⏱️ Patrón 3: Timeout

Configuración: 30 segundos máximo por request

Beneficios: Previene requests colgadas, evita thread pool exhaustion

🔗 Combinación (Wrap Order)

Timeout (30s total) └─ Circuit Breaker (5 failures = open) └─ Retry (3 intentos con backoff) └─ HTTP Call
✓ Status: 3 patrones implementados y funcionales

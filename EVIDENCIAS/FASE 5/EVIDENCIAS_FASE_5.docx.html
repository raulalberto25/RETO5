ADR-001: Accounts como Primer Módulo a Extraer

Estado: ✅ Aprobado
Contexto: ¿Cuál módulo extraer primero?
Opciones: Accounts (ELEGIDA), Transfers, Notifications, Audit, Auth
Decisión: Accounts por autonomía de dominio, base de todas operaciones, low risk
Consecuencias: ✅ Establece patrón | ❌ Requiere nueva BD

ADR-002: Transfers como Segundo Módulo a Extraer

Estado: ✅ Aprobado
Contexto: Con Accounts MS operativo, ¿cuál es el siguiente?
Opciones: Transfers (ELEGIDA), Notifications, Audit
Decisión: Transfers por flujo crítico, depende de Accounts, requiere async
Consecuencias: ✅ Demuestra Saga + Outbox | ❌ Complejidad significativa

ADR-003: API Gateway (YARP)

Estado: ✅ Aprobado
Contexto: Reverse proxy para rutear a MS/Monolito
Opciones: YARP (ELEGIDA), Nginx, Kong, AWS ALB
Decisión: YARP por nativo .NET, bajo overhead, despliegue trivial
Consecuencias: ✅ Stack homogéneo | ❌ Menos probado que Nginx

ADR-004: Message Broker (RabbitMQ)

Estado: ✅ Aprobado
Contexto: Broker para comunicación asincrónica
Opciones: RabbitMQ (ELEGIDA), Kafka, AWS SNS/SQS
Decisión: RabbitMQ por durabilidad ACID, replay, baja complejidad operacional
Consecuencias: ✅ Garantía de entrega | ❌ Si > 100k msg/sec, Kafka mejor

ADR-005: BD Accounts MS (PostgreSQL)

Estado: ✅ Aprobado
Contexto: BD exclusiva para Accounts MS
Opciones: PostgreSQL (ELEGIDA), MongoDB, Cassandra
Decisión: PostgreSQL por ACID, mismo motor que monolito, schema aislado
Consecuencias: ✅ Transacciones ACID | ❌ Overhead > NoSQL

ADR-006: BD Transfers MS (PostgreSQL)

Estado: ✅ Aprobado
Contexto: BD exclusiva para Transfers MS con tabla outbox
Opciones: PostgreSQL (ELEGIDA), MongoDB
Decisión: PostgreSQL por transacciones atómicas, Outbox pattern nativo
Consecuencias: ✅ Outbox nativo | ❌ 3 instancias PostgreSQL

ADR-007: Patrón Consistencia Distribuida (Saga Choreography)

Estado: ✅ Aprobado
Contexto: Operación atómica local → distribuida async
Opciones: Saga Choreography (ELEGIDA), Saga Orchestrator, Strong Consistency
Decisión: Saga Choreography + Eventual Consistency por desacoplamiento, sin SPOF
Consecuencias: ✅ Escalable, desacoplado | ❌ Eventual consistency, debug difícil

ADR-008: Arquitectura Interna (Hexagonal)

Estado: ✅ Aprobado
Contexto: Estructura interna de cada microservicio
Opciones: Hexagonal (ELEGIDA), Simple Layers, CQRS
Decisión: Hexagonal (Ports & Adapters) por testabilidad, DDD-aligned
Consecuencias: ✅ Testeable, flexible | ❌ Más código, curva aprendizaje

ADR-009: Stack Observabilidad (OpenTelemetry)

Estado: ✅ Aprobado
Contexto: Trazas, métricas, logs
Opciones: OpenTelemetry (ELEGIDA), Datadog, New Relic
Decisión: OpenTelemetry → Jaeger + Prometheus + Grafana + Loki (vendor-agnostic)
Consecuencias: ✅ Open source, flexible | ❌ Múltiples componentes

ADR-010: Contrato de Eventos (CloudEvents 1.0)

Estado: ✅ Aprobado
Contexto: Formato y versionamiento de eventos
Opciones: CloudEvents 1.0 (ELEGIDA), JSON plano, Avro
Decisión: CloudEvents 1.0 + JSON Schema por standard CNCF, versionable, traceable
Consecuencias: ✅ Portable, standard | ❌ Overhead de headers

ADR-011: Migración Zero-Downtime (Dual-write)

Estado: ✅ Aprobado
Contexto: Migrar datos sin downtime
Opciones: Dual-write + Cutover (ELEGIDA), Bulk copy, Replication
Decisión: Dual-write → Backfill → Read Switchover → Stop Dual-write
Consecuencias: ✅ Zero downtime, reversible | ❌ Complejidad temporal

⚖️ Análisis de Trade-offs

P1: ¿Consistencia Eventual Aceptable para Operaciones Bancarias?

Respuesta: NO universalmente, SÍ con excepciones (90% de casos)

Aceptable: Transferencias (core strong), Notifications/Audit (eventual)
NO Aceptable: Verificar balance, Prevención doble gasto, Límites diarios

Solución: Transfer (strong local) + side effects (eventual) ✅
P2: ¿Operaciones Sacrificando Disponibilidad para Consistencia?

Respuesta: SÍ, 3 casos críticos

Caso 1 - Verificación: Circuit breaker si Accounts MS cae → 503 Unavailable
Caso 2 - Debit/Credit: Eventual consistency + compensación si falla
Caso 3 - Concurrencia: Riesgo de race conditions (bajo volumen actual)

Mitigación: Aceptables porque probabilidad baja + volumen bajo
P3: ¿Reversibilidad de Decisiones?

Reversibles (< 1 semana): ADR-001, 003, 008, 009, 011
Parcialmente (1-4 semanas): ADR-004, 007, 010
NO Reversibles (> 4 semanas): ADR-005 (Accounts DB), ADR-006 (Transfers DB)

Lección: Database-per-Service es decisión CRÍTICA (no volver atrás)
P4: ¿Complejidad Operativa Justificada?

Costos Reales: Infraestructura 3x, DevOps +0.5 FTE, debugging distribuido

Beneficios: Escalabilidad 3x, équipos paralelos, resiliencia graceful, observabilidad 10x

Conclusión: ✅ Justificado para futuro (volumen creciente), ⚠️ NO plenamente para presente (bajo volumen)
P5: ¿Módulos NO Extraídos? ¿Por Qué?

Extraídos: Accounts (core autónomo), Transfers (core critical)
NO extraídos: Auth (transversal), Notifications (side effect), Audit (historical)

Patrones Alternativos:
- Auth: Middleware en Gateway
- Notifications/Audit: Consumer async en monolito

Lección: Microservicios ≠ extraer TODO. Cada módulo tiene patrón óptimo
